The differentiator

Memory & rules engine

40+ memory files encode how every article gets written — loaded automatically by /article. This is what keeps AI-assisted output on-brand and indistinguishable from hand-writing. A sample of the governing rules:

✍️ Voice & writing

  • Human-indistinguishable, AI-detector safe; UK spelling, contractions, zero prose em dashes
  • No AI filler ("it's worth noting", "furthermore", "when it comes to")
  • Strip trader/finance jargon from body copy
  • Travel/food = "sharp friend who's done the trip" voice
  • Every delivery includes a fun one-liner + 3–4 alternatives

🧱 Structure & format

  • Answer-first universal: answer block adjacent to H1, then TLDR table (the answer block)
  • HTML TOC anchors, title-case H2s, dense paras → bullets
  • Currency: 2,990 JPY (~S$27) — code after number, no symbols
  • Product launches + scam advisories = their own formats
  • Conclusion = punchy one-liner + YTBLOG5 CTA blurb

⚖️ Finance framing

  • Honest reviewer who happens to work for YouTrip: neutral body, opinionated verdict
  • YouTrip = best for the specific job, never "best overall"
  • Never recommend or link Wise / Revolut / competitors
  • Plain language ban-list; let the maths land in real dollars
  • Payment hierarchy: tap YouTrip card → cash → TNG backup

🔎 Fact-checking

  • Verify all numbers proactively — never wait to be asked
  • STRICT (finance, official sources) vs STANDARD (travel) auto-detected
  • Treat /research output + the source article as claims, not facts
  • Never flag-and-leave unverified numbers — rephrase, range, or remove
  • ATM cash via YouTrip, never money changers

Backed by shared/decisions.md (rationale log), shared/workflows.md (SEO/AEO/GEO master reference + on-page checklist), fact_ledger.md, and retrospectives/. Conflict rule: latest SEO best practice beats a previously-saved voice rule.

The honest read

Assessment & what to improve next

The build phase is genuinely impressive — the research → write → fact-check engine is real and running. But the data shows the value is leaking at one specific point, and a few high-leverage gaps remain. Prioritised below.

What's working
✓ End-to-end content engine (init 1–3) genuinely done ✓ Memory rules keep AI output on-brand ✓ News + FX + core-update monitoring all live ✓ Two-mode fact-checking kills round-to-round drift ✓ WordPress no-dev publish path validated ✓ Strategy already rebalanced toward defenders
PRIORITY 1

Close the publish loop — 10 drafts, 0 live

The factory produced 10 drafts; the ledger says none are published and 5 have sat ≥16 days. Every hour of research, drafting and fact-checking upstream is stranded until these ship. This is the single biggest ROI move available right now.

Do: batch-publish the backlog through wp_publish.py, record each live URL via pipeline_ledger.py published, and wire the publish step as the explicit final stage of /factory so new drafts don't pool. Why it matters: nothing downstream (attribution, days-to-publish, scanner-vs-handbrief comparison) can be measured until URLs are live.

PRIORITY 1

Unblock Semrush units

The NEW/gap stream of the keyword scanner and /competitor-gap are both throttled by a depleted Semrush units balance (ERROR 132). Two of your discovery tools are running at partial power.

Do: the cheapest unblock per your own roadmap is refactoring pull_semrush_gaps() from domain_domains (80 units/row) to domain_organic (10/row) and computing the gap in Python — 8× cheaper, no spend. The ~US$50 top-up is the fallback. Note: the semrush-server MCP is on a separate balance and still works — don't conflate them.

PRIORITY 2

Build Initiative 6 (rank-driven refresh detection) next, not net-new

Most YouTrip articles are already published, and your own 13 May strategy decision shifted effort toward defender maintenance over publishing volume. Initiative 6 is the automation that serves that strategy — yet it's at 0%.

Do: weekly GSC scan for pages dropping 3+ positions over 2 weeks → auto-refresh brief. You already have /update (manual) and the surgical-edit tiering rule — Init 6 is the trigger layer on top. Protects an estimated +800–1,500 clicks/mo. Higher leverage than Init 7/9 given the lag math.

PRIORITY 2

Add a "which command when" router

21 skills is powerful but several overlap: /weekly vs /seo-analysis vs /track-targets vs /monthly-report all answer "how are we doing"; /keyword-scan vs /competitor-gap vs /quick-wins all surface opportunities. Friction is choosing.

Do: a one-page decision guide (daily / weekly / monthly / ad-hoc, "I want to find topics" vs "I want to report") — could even be a section in this page or a /help-style skill. Low effort, removes a recurring micro-decision.

PRIORITY 3

Add the GA4 engagement layer

You measure clicks (GSC) but not what happens after the click. The roadmap's GA4 MCP server (Phase 3) is unbuilt, so "which articles drive real engagement, not just traffic" is still a blind spot.

Do: build ~/ga4-mcp/ on the same OAuth pattern as GSC. Lets /monthly-report and /seo-analysis distinguish high-traffic-low-engagement pages — directly informs the refresh queue.

PRIORITY 3

Actually run the measurement loop

Every initiative has named target metrics (edit-acceptance ≥70%, topics-surfaced/week, days-to-publish, scanner-sourced vs hand-briefed clicks) — but with 0 articles published, almost none are being tracked yet. The instrumentation exists; the data doesn't flow.

Do: once P1 ships URLs, schedule a monthly read of the ledger metrics so you can prove the engine's ROI to leadership — which is exactly the credibility-with-sourced-numbers bar you hold yourself to.

One-line verdict

You've over-built the creation half and under-built the distribution + measurement half. The fastest win isn't another tool — it's shipping the 10 drafts already sitting in the outbox, then turning the maintenance automations (Init 6) on so the system protects traffic it's already earned.
